框架组成与各自职责

Spring - 容器框架，团队的“大管家”

  • SSM 框架的核心和基石。它就像一个大型的“集装箱”或“大管家”，负责整合和管理所有其他组件。

  • 核心功能：

    IoC (控制反转)： 将对象的创建、依赖关系的维护（“谁使用谁”）的权利从程序员手中“反转”给 Spring 容器来管理。这极大地降低了程序组件之间的耦合度。开发者通过配置文件或注解定义 Bean，Spring 负责实例化并组装它们。

    AOP (面向切面编程)： 将那些遍布在业务逻辑中的通用功能（如日志记录、事务管理、安全校验、性能监控等）抽取出来，形成独立的“切面”。然后在这些业务逻辑的执行过程中，在合适的切入点动态地植入这些功能。这使得业务代码更加纯净，只关注核心逻辑。

    声明式事务： 这是 AOP 的一个典型应用。开发者只需通过简单的配置（如 @Transactional 注解）来管理事务，无需在代码中手动编写繁琐的 beginTransaction(), commit(), rollback() 等代码。Spring 会为我们自动完成。

Spring MVC - MVC 框架，网站的“调度中心”

  • 基于 Spring 的 Web 层框架，负责处理用户的请求并给出响应。它实现了经典的 MVC (Model-View-Controller) 设计模式，是网站的“调度中心”。

  • 工作原理：

    Controller (控制器)： 接收用户请求（如浏览器发送的 HTTP 请求），调用 Service 层处理业务逻辑，并返回一个模型和视图（ModelAndView）或直接返回数据（如 JSON）。

    Model (模型)： 封装了业务数据，通常就是 Java 对象（POJO），在控制器和视图之间传递。

    View (视图)： 负责渲染模型数据，展示给用户。通常是 JSP, Thymeleaf, Freemarker 等模板技术，现在更流行前后端分离，返回 JSON 数据，由前端框架（如 Vue, React）来渲染页面。

  • 核心组件： DispatcherServlet（前端控制器，所有请求的入口）、HandlerMapping、Controller、ViewResolver 等。

MyBatis - 持久层框架，数据库的“操作员”

  • 负责与数据库打交道，完成数据的增删改查（CRUD）操作。它是团队的“数据库操作员”。

  • 核心功能：

    SQL 与代码分离： 将 SQL 语句写在 XML 配置文件或注解中，与 Java 代码解耦，便于维护和优化 SQL。

    ORM 映射： 自动将 JDBC 返回的 ResultSet 结果集映射成 Java 对象，省去了手动封装对象的繁琐过程。

    动态 SQL： 提供了强大的标签（如 , ），可以根据条件动态拼接 SQL 语句，非常灵活。

SSM 框架的协同工作流程（核心）

SSM 框架的精华在于它们之间的分工与协作。一个典型的 HTTP 请求处理流程如下：

  1. 用户发起请求： 浏览器向服务器发送一个 HTTP 请求（例如：/user/1）。

  2. Spring MVC 接收（DispatcherServlet）： 请求首先被 DispatcherServlet 拦截，它是整个 MVC 模式的调度中心。

  3. Spring MVC 路由（HandlerMapping）： DispatcherServlet 查询 HandlerMapping，找到能处理该请求的 Controller 中的具体方法。

  4. Controller 处理请求：

  • Controller 方法接收请求参数。

  • 调用 Service 接口（业务层）的方法进行业务处理。

  1. Spring 依赖注入（IoC）： Spring 容器会自动将 Service 接口的实现类（ServiceImpl）注入（@Autowired）到 Controller 中。Service 实现类中又会自动注入 Mapper 接口（@Autowired）。

  2. Service 处理业务逻辑： Service 层包含复杂的业务逻辑，它负责协调多个数据库操作，并通过 Spring 的事务管理保证数据一致性。

  3. MyBatis 操作数据库：

  • Service 调用 Mapper 接口的方法。

  • MyBatis 在运行时会根据接口方法名，找到对应的 Mapper.xml 文件中配置的 SQL 语句并执行。

  • MyBatis 将 SQL 执行结果映射成 Java 对象并返回。

  1. 返回结果：
  • Service 将处理结果返回给 Controller。
  • Controller 将结果数据（Model）封装起来，并指定一个视图名称（或直接返回 JSON 数据）。

  1. 视图解析（ViewResolver）： DispatcherServlet 调用 ViewResolver 来解析视图，找到具体的视图页面（如 JSP）。

  2. 渲染视图： 视图页面使用模型数据（Model）进行渲染，生成最终的 HTML 页面。

  3. HTTP 响应： DispatcherServlet 将渲染结果返回给浏览器，用户看到响应页面。

SSM 框架的特点与现状

特点：

  • 层次清晰： 各层之间职责单一，耦合度低，易于开发和维护。

  • 高度可配置： 整合方式灵活，可以通过 XML 或注解进行配置。

  • 生态强大： 三个框架都非常成熟稳定，社区活跃，资源丰富。

  • 灵活性高： 尤其是 MyBatis，对 SQL 的可控性使其能适应复杂业务和高性能要求。
