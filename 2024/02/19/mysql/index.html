数据库

数据库全称 DataBase，简称 DB。是安装在操作系统之上的数据仓库软件，可以原来存储数据，管理数据

数据库分类

关系型数据库：SQL（Structured Query Language）

  • MySQL、Oracle、Sql Server、DB2、SQLlite、PostgreSQL
  • 通过表和表之间，行和列之间的关系进行数据的存储
  • 通过外键关联来建立表与表之间的关系

非关系型数据库：NoSQL（Not Only SQL）

  • Redis、MongoDB
  • 指数据以对象的形式存储在数据库中，而对象之间的关系通过每个对象自身的属性来决定

MySQL

MySQL 搭建

了解如何快速部署一个 MySQL 环境对于开发和测试至关重要

主要方式：

  • 本地安装：从 MySQL 官网 下载安装包，在本地操作系统（Windows, macOS, Linux）上安装。

  • 使用 Docker：这是目前最干净、最便捷的方式，可以避免污染本地环境。

SQL 语句

SQL 是与数据库交互的语言，分为以下几类

  1. DDL - 数据定义语言 (操作数据库和表结构)
  1. DML - 数据操作语言 (操作数据本身)
  1. DQL - 数据查询语言 (主要指复杂的 SELECT 查询)

索引

索引是帮助 MySQL 高效获取数据的数据结构（通常是 B+Tree），类似于书的目录。

  • 优点：大大加快数据的检索速度。

  • 缺点：降低增、删、改的速度（因为索引也需要维护），并占用额外空间。

何时创建索引：

  • 经常作为查询条件（WHERE）的字段。

  • 经常需要排序（ORDER BY）或分组（GROUP BY）的字段。

  • 用于表连接（JOIN）的字段（外键）。

事务

事务是一组要么全部成功、要么全部失败的 SQL 操作。它必须满足 ACID 特性：

  • 原子性 (Atomicity)：事务中的所有操作是一个不可分割的整体。

  • 一致性 (Consistency)：事务使数据库从一个一致状态转变到另一个一致状态。

  • 隔离性 (Isolation)：并发事务之间相互隔离，互不干扰。

  • 持久性 (Durability)：事务一旦提交，其对数据的修改就是永久性的。

锁机制

锁是数据库处理并发访问的机制，保证数据的一致性

按粒度分：

  • 表锁：开销小，加锁快，锁定整张表，并发度低。

  • 行锁：开销大，加锁慢，只锁定需要操作的行，并发度高。（InnoDB 引擎支持）

按行为分：

  • 共享锁 (S 锁)：又称读锁。其他事务可以加共享锁读，但不能加排他锁写。
  • 排他锁 (X 锁)：又称写锁。其他事务不能加任何锁

悲观锁与乐观锁

  • 悲观锁：假定会发生并发冲突，先上锁再访问（SELECT … FOR UPDATE）。

  • 乐观锁：假定不会发生冲突，提交更新时再检查。通常通过版本号或时间戳实现。

设计数据库表

范式化 (Normalization)

  • 第一范式 (1NF)：列不可再分，确保每列原子性。

  • 第二范式 (2NF)：在 1NF 基础上，非主键列必须完全依赖于整个主键（针对联合主键）。

  • 第三范式 (3NF)：在 2NF 基础上，非主键列之间不能有传递依赖。

遵循范式的优点：减少数据冗余，避免更新异常。

反范式化的优点：减少表连接，提高查询性能。这是一种以空间换时间的权衡。

设计步骤与考量

  • 理解需求：明确业务实体和它们之间的关系。

  • 定义实体和属性：确定有哪些表，每个表有哪些字段。

  • 选择主键：优先使用无业务意义的自增 ID（代理主键）还是有业务意义的字段（自然主键）？

  • 建立关系：

    1. 一对一：users 和 user_profiles，可以用相同的主键。

    2. 一对多：users 和 orders，在“多”的一方（orders）加外键 user_id。

    3. 多对多：students 和 courses，需要第三张关联表 student_courses，包含两个外键 student_id 和 course_id。

  • 选择数据类型：在满足需求的前提下，选择尽可能小的数据类型（如 TINYINT vs INT）。

  • 考虑可扩展性
